## Context

The existing Instant Share system allows iOS-to-macOS sharing via Share Extension → mDNS discovery → DH trust handshake → payload upload. This change inverts the direction: **macOS-to-iOS sharing**, triggered from the macOS right-click/share menu.

Current state:
- **Launch Agent** (InstantShareAgent) already runs an HTTP server on port `9527` with trust and transfer endpoints
- **iOS AuBackup** already has QR code scanning for backup pairing and an `InstantShareUploadClient` for uploading payloads
- **No macOS Share Extension** exists — only an iOS one

The inversion requires a new share trigger on macOS, a simplified trust model (opt-code via QR instead of DH + PIN), a Unix domain socket for local extension↔agent communication, and a pull-based transfer where iOS downloads from the PC.

## Goals / Non-Goals

**Goals:**
- macOS Share Extension (native Swift) that appears in the system share menu for selected files and text
- Share Extension sends payload to the Launch Agent via HTTP over Unix domain socket
- Launch Agent displays a QR code mini-window encoding PC IPs + port + opt code
- AuBackup iOS app scans QR, pulls the payload from the Launch Agent's TCP HTTP server, and presents text or saves image to photo library
- Security: opt-code prevents unauthorized access; single-use codes with TTL

**Non-Goals:**
- Not replacing the existing iOS Share Extension (both coexist)
- Not adding mDNS/BLE discovery for this flow (QR replaces discovery)
- Not supporting batch/album transfers (single file or text per share)
- Not adding TLS to LAN traffic (same trust model as existing instant share — app-layer security)

## Decisions

### Decision 1: macOS Share Extension via native Swift

**Choice**: Native Swift Share Extension using AppKit/SwiftUI, built as a separate target bundled inside AuSearch.app.

**Rationale**: Native Swift gives the best integration with the macOS share menu, lowest overhead, and direct access to `NSExtension` APIs. The extension process is lightweight — it receives the payload, sends it to the Launch Agent via Unix socket, and exits. No persistent UI is shown by the extension; the Launch Agent owns the user-facing QR window.

**Alternatives considered**: Python/PyObjC extension (higher overhead, more complex sandboxing), AppKit `NSSharingService` subclass (more limited activation UX).

### Decision 2: Opt-code via QR (not DH handshake)

**Choice**: A lightweight opt-code (6-digit single-use code with 5-minute TTL) embedded in the QR alongside PC IPs and port. No DH key exchange or PIN verification needed.

**Rationale**: 
- The share extension already runs on the same machine as the Launch Agent, so the local communication is trusted (Unix domain socket)
- The QR is shown on the PC screen and scanned by the user's own phone — physical proximity is the trust mechanism
- No need for the 3-round-trip DH handshake since the opt-code is generated by the same process that receives the payload
- Simpler, faster, fewer failure modes

**Alternatives considered**: Full DH handshake + PIN (unnecessary complexity for same-machine share extension → launch agent, adds latency), No opt-code (allows anyone on the LAN to claim payloads), Encrypted QR (adds complexity with no real benefit — QR is scanned from the PC screen by the user's own camera).

### Decision 3: Pull-based transfer (PC hosts, iOS pulls)

**Choice**: The Launch Agent hosts the payload as a temporary resource. iOS discovers the PC via QR and issues an HTTP POST to download. For text, the payload is buffered in memory; for images, the agent streams the file from the original path.

**Rationale**: The existing Launch Agent already runs an HTTP server. Inverting to push (PC → iOS without request) would require the iOS app to run a listener — which it cannot do reliably in background. Pull is simpler and matches the "scan QR to claim" mental model.

**Alternatives considered**: Push via WebSocket (too complex for a single payload), Notification-based (requires APNs setup, intermittent), Push via mDNS reverse (creates unnecessary coupling with existing discovery).

### Decision 4: Endpoint structure `/api/instant-share/v1/qr-trigger` and `/api/instant-share/v1/qr-claim`

**Choice**: Two distinct top-level paths under the v1 namespace — one for triggering the share from the macOS extension, one for claiming it from the mobile client.

**Rationale**: The trigger path lives under the `qr-trigger` namespace (extension-specific). The claim path lives directly under `qr-claim` since it's a public-facing API consumed by a different client (iOS). This keeps the trigger internals separated from the public claim surface.

**Endpoints:**
- `POST /api/instant-share/v1/qr-trigger` — macOS Extension → Agent (Unix socket only)
- `POST /api/instant-share/v1/qr-claim` — iOS → Agent (TCP, LAN)

### Decision 5: Unix domain socket inside extension's sandbox container

**Choice**: The macOS Share Extension sends the stash request over HTTP via a Unix domain socket, not TCP localhost. The Launch Agent creates the socket **inside the extension's sandbox container** so the sandboxed extension can reach it. The agent is not sandboxed and can write into the extension's container directory.

**Rationale**: 
- The extension is sandboxed — it can only connect to sockets within its own container
- Unix domain sockets have lower overhead than TCP loopback
- No port conflict risk with the existing port 9527 TCP server
- No sandbox network entitlements needed for the extension
- The agent knows the extension's bundle ID and can construct the container path: `~/Library/Containers/<share-extension-bundle-id>/Data/Library/Application Support/au-search/qr-transfer.sock`

**Alternatives considered**: TCP localhost (needs network entitlements, port conflict risk), App group container (adds app group configuration overhead), Named pipe (less idiomatic on macOS).

### Decision 6: Image file path (not bytes) from extension to agent

**Choice**: For image shares, the extension sends the absolute file path to the agent instead of raw image bytes. The agent reads the file directly. For text shares, the extension sends the text content inline.

**Rationale**: 
- The Launch Agent runs as the same user and is not sandboxed — it can read any file the user can access
- Avoids copying large image data through the IPC channel
- No 50MB memory limit needed for the stash — the agent can stream the file to the iOS client

**Alternatives considered**: Sending raw bytes (wastes memory, slower for large files), File sandboxing workaround (more complex).

## Risks / Trade-offs

- **[Risk] Share Extension sandboxing**: macOS sandbox restricts filesystem access. **Mitigation**: Agent creates the socket inside the extension's sandbox container directory — the extension can always reach paths within its own container. The agent constructs the path from the known bundle ID.
- **[Risk] File path validity window**: The file could be moved or deleted between the extension sending the path and the agent reading it. **Mitigation**: For short-lived flows (seconds), this is unlikely. The agent reads the file immediately upon receiving the stash request.
- **[Risk] Opt-code collision**: With 6 digits (1M combinations), collision is possible if multiple shares happen in quick succession. **Mitigation**: Add stash_id (UUID) to QR so the opt-code only gates on the correct stash; opt-code is scoped to a specific stash.
- **[Risk] iOS app in background**: User may scan QR while iOS app is in background — notification/badge to bring user back. **Mitigation**: Use URL scheme (`aubackup://qr-claim?data=...`) to deep-link into AuBackup from the QR scan.
- **[Risk] Unix socket path coordination**: The extension's sandbox container path depends on its bundle ID. **Mitigation**: The bundle ID is a build-time constant known to both the extension and the agent. The agent constructs the path and creates the socket there.

## Open Questions

- What is the exact sandbox container path for the share extension? Derived from bundle ID convention: `~/Library/Containers/<bundle-id>/Data/Library/Application Support/au-search/qr-transfer.sock`.
- What is the UX for the iOS app when multiple PCs are on the same LAN but only one has a pending share? The QR encodes the specific PC IP and port, so there's no ambiguity.
- Should the opt-code expire on first unsuccessful attempt (to prevent brute-force)? Yes — 3 failed attempts invalidate the code.
