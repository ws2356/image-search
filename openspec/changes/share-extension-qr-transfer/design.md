## Context

The existing Instant Share system allows iOS-to-macOS sharing via Share Extension → mDNS discovery → DH trust handshake → payload upload. This change inverts the direction: **macOS-to-iOS sharing**, triggered from the macOS right-click/share menu.

Current state:
- **Launch Agent** (InstantShareAgent) already runs an HTTP server on port `9527` with trust and transfer endpoints
- **iOS AuBackup** already has QR code scanning for backup pairing and an `InstantShareUploadClient` for uploading payloads
- **No macOS Share Extension** exists — only an iOS one

The inversion requires a new share trigger on macOS, a simplified trust model (opt-code via QR instead of DH + PIN), and a pull-based transfer where iOS downloads from the PC.

## Goals / Non-Goals

**Goals:**
- macOS Share Extension (NSExtension) that appears in the system share menu for selected files and text
- Share Extension sends payload to the Launch Agent via HTTP POST
- Launch Agent displays a QR code mini-window encoding PC IPs + opt code
- AuBackup iOS app scans QR, pulls the payload from the Launch Agent, and presents text or saves image to photo library
- Security: opt-code prevents unauthorized access; single-use codes with TTL

**Non-Goals:**
- Not replacing the existing iOS Share Extension (both coexist)
- Not adding mDNS/BLE discovery for this flow (QR replaces discovery)
- Not supporting batch/album transfers (single file or text per share)
- Not adding TLS to LAN traffic (same trust model as existing instant share — app-layer security)

## Decisions

### Decision 1: NSExtension Share Extension (not AppKit Service)

**Choice**: macOS Share Extension using `NSExtension` with `NSExtensionPointIdentifier = com.apple.share-services`.

**Rationale**: Follows the same pattern as the existing iOS Share Extension, appears in the standard Share menu, gets its own process sandbox, and is the officially supported way to add share targets on macOS. An AppKit Service would have different activation UX and no dedicated UI lifecycle.

**Alternatives considered**: AppKit `NSSharingService` subclass (more limited, harder to show custom UI), Quick Action (only for Finder files, not text).

### Decision 2: Opt-code via QR (not DH handshake)

**Choice**: A lightweight opt-code (6-digit single-use code with 5-minute TTL) embedded in the QR alongside PC IPs. No DH key exchange or PIN verification needed.

**Rationale**: 
- The share extension already runs on the same machine as the Launch Agent, so the local communication is trusted (localhost)
- The QR is shown on the PC screen and scanned by the user's own phone — physical proximity is the trust mechanism
- No need for the 3-round-trip DH handshake since the opt-code is generated by the same process that receives the payload
- Simpler, faster, fewer failure modes

**Alternatives considered**: Full DH handshake + PIN (unnecessary complexity for same-machine share extension → launch agent, adds latency), No opt-code (allows anyone on the LAN to claim payloads), Encrypted QR (adds complexity with no real benefit — QR is scanned from the PC screen by the user's own camera).

### Decision 3: Pull-based transfer (PC hosts, iOS pulls)

**Choice**: The Launch Agent hosts the payload as a temporary resource. iOS discovers the PC via QR and issues an HTTP GET/POST to download. The payload is buffered in memory on the Launch Agent.

**Rationale**: The existing Launch Agent already runs an HTTP server. Inverting to push (PC → iOS without request) would require the iOS app to run a listener — which it cannot do reliably in background. Pull is simpler and matches the "scan QR to claim" mental model.

**Alternatives considered**: Push via WebSocket (too complex for a single payload), Notification-based (requires APNs setup, intermittent), Push via mDNS reverse (creates unnecessary coupling with existing discovery).

### Decision 4: New endpoint prefix (`/api/instant-share-v2/` or `/api/qr-transfer/`)

**Choice**: New endpoints under `/api/qr-transfer/v1/` to keep them separate from the existing instant share v1 protocol.

**Rationale**: The v1 endpoints (`/api/instant-share/v1/`) expect the full trust handshake flow. Mixing the opt-code flow would add complexity to handler logic. A separate prefix keeps concerns clean.

### Decision 5: Share Extension communicates with Launch Agent via localhost

**Choice**: The Share Extension sends the payload to `http://127.0.0.1:9527/api/qr-transfer/v1/stash`. The Launch Agent listens on localhost (plus LAN IPs for iOS downloads) but the stash endpoint only accepts from localhost.

**Rationale**: The Share Extension runs as a separate sandboxed process. The Launch Agent is the natural rendezvous point. Localhost-only for stashing prevents remote machines from pushing payloads.

## Risks / Trade-offs

- **[Risk] Share Extension sandboxing**: macOS sandbox may restrict network access to localhost. Verify `com.apple.security.network.local` entitlement.
- **[Risk] Launch Agent port conflict**: Port 9527 is already used. Must verify the QR claim handler registers on the same server or a separate port. **Mitigation**: Add new routes to the existing `InstantShareHTTPServer` under the new prefix.
- **[Risk] Opt-code collision**: With 6 digits (1M combinations), collision is possible if multiple shares happen in quick succession. **Mitigation**: Add stash_id (UUID) to QR so the opt-code only gate on the correct stash; opt-code is scoped to a specific stash.
- **[Risk] iOS app in background**: User may scan QR while iOS app is in background — notification/badge to bring user back. **Mitigation**: Use URL scheme (`aubackup://qr-claim?data=...`) to deep-link into AuBackup from the QR scan.
- **[Risk] Large files**: Loading large files into memory on the Launch Agent could exhaust resources. **Mitigation**: Cap at 50MB for stashing; larger files show an error. The use case is quick shares (screenshots, documents, text snippets), not media libraries.

## Open Questions

- Should the QR include the stash_id in the URL, or is opt-code enough? Recommendation: Include both — the stash_id scopes the opt-code.
- What is the UX for the iOS app when multiple PCs are on the same LAN but only one has a pending share? The QR encodes the specific PC IP, so there's no ambiguity.
- Should the opt-code expire on first unsuccessful attempt (to prevent brute-force)? Yes — 3 failed attempts invalidate the code.
