# Instant Sharing Protocol API Spec

## 1. Scope

This document defines the dedicated instant-sharing protocol contracts for mDNS (Bonjour) discovery, session bootstrap, trust establishment, and payload transfer. The protocol is independent from QR backup session endpoints and MUST NOT depend on backup-session capability exchange.

Goals covered:
- Discover candidate PCs via desktop mDNS daemon service advertisement.
- Bootstrap transport with mobile-selected session ID plus mobile IP/port list over HTTP callback to PC.
- Establish first-time trust using DH key exchange, encrypted PIN apply, and confirm exchange.
- Transfer payloads over HTTPS using pinned trust and request authentication based on session-id signature verification.
- Enforce single active receive session per desktop receiver.

Out of scope:
- Backup-session APIs.
- Large-media optimization strategies.
- Concurrent instant-share transfers.

## 2. Transport, Versioning, and Session Bootstrap

- Base scheme: HTTPS.
- API prefix: `/api/instant-share/v1`.
- Control plane content type: `application/json`.
- Binary transfer: `multipart/form-data`.
- Transport authentication:
  - First share uses the DH/PIN trust flow to establish long-term trust material.
  - Subsequent shares MUST include a signature of `X-Session-Id` generated by PC private key.
  - Mobile MUST verify that signature using the previously exchanged/stored trusted PC public key.
- Session bootstrap rule:
  - The Share Extension presents the mDNS-discovered device list and opens AuBackup when the user selects a PC.
  - Session is created in AuBackup after it resumes selected device and payload context from the Share Extension handoff.
  - AuBackup sends an HTTP POST to the PC's bootstrap endpoint (resolved IP:port from mDNS) to deliver session bootstrap data (mobile IP list, mobile port, session ID).
  - PC MUST include `X-Session-Id` in all HTTP calls to mobile.
  - Mobile MUST validate incoming session ID against the active session generated from the Share Extension handoff context.
- Role direction in v1:
  - Mobile hosts the instant-sharing HTTP service using the IP/port from the session bootstrap.
  - PC acts as HTTP caller for trust and payload-download operations.

Versioning:
- Backward-compatible changes: additive optional fields.
- Breaking changes: new major path (`/v2`).

## 3. Identity and Metadata

### 3.1 Required HTTP Headers

- `X-Instant-Share-Version`: protocol version (e.g., `1.0`).
- `X-Correlation-Id`: UUID generated per share attempt.
- `X-Session-Id`: UUID generated by AuBackup after resuming Share Extension handoff context.
- `X-Device-Id`: caller device identifier.
- `X-Session-Signature`: base64 signature of `X-Session-Id` using caller private key.
- `X-Session-Signature-Alg`: signature algorithm identifier (for example `ed25519`).

### 3.2 Required Metadata Fields

- `flow_id`: fixed value `instant_share`.
- `payload_class`: current implementation slice supports `text | image`.
- `target_intent`:
  - `text`: `clipboard_only`
  - `image`: `clipboard_or_file`
- `trust_mode`: `first_share | trusted_direct`.

## 4. mDNS Advertisement Contract (Desktop Daemon)

Desktop daemon MUST advertise one Bonjour service for discovery.

Service:
- Type: `_instantshare._tcp`
- Domain: `local`
- Name: human-readable device name (e.g. `My Mac`)
- Port: PC's instant-share HTTP API port

TXT Records:

| Key | Value | Description |
|-----|-------|-------------|
| `ver` | string | Protocol version (e.g. `1`) |
| `device_name` | string | Human-readable PC display name |
| `device_id` | string | Persistent unique device identifier (UUID) |
| `signature` | base64 | Cryptographic signature for trusted-direct verification |
| `signature_key_id` | string | Key identifier for signature verification |
| `timestamp_ms` | integer string | Unix millisecond timestamp for signature freshness |

Rules:
- PC MUST update `signature` and `timestamp_ms` periodically.
- Mobile MUST verify `signature` against pinned trusted public key before trusted-direct share.
- If signature verification fails, mobile MUST fall back to full first-share trust flow.

## 4b. Session Bootstrap Endpoint (PC HTTP API)

After mDNS discovery, mobile bootstraps the session by posting to the PC's HTTP API.

### POST /api/instant-share/v1/sessions/bootstrap
Sends session bootstrap data from mobile to desktop receiver.

Request:
- `session_id`: UUID generated by AuBackup
- `mobile_port`: integer (mobile HTTPS server port)
- `mobile_ip_list`: array of IP strings (IPv4/IPv6 allowed)
- `correlation_id`: UUID
- `payload_class`: `text | image`
- `target_intent`: `clipboard_only | clipboard_or_file`

Response 200:
- `accepted`: true
- `pc_device_id`: string (for mobile verification — must match mDNS TXT `device_id`)

Rules:
- PC MUST persist bootstrap data for the active session window.
- PC MUST reject HTTP requests whose session ID does not match current active bootstrap.
- PC MUST reject bootstrap when another session is active (return `RECEIVER_BUSY_SINGLE_SESSION`).
- Mobile MUST verify response `pc_device_id` matches the `device_id` from the mDNS TXT record of the selected device.

## 5. Trust Endpoints

## 5.1 POST /api/instant-share/v1/trust/handshake
Performs DH handshake and returns material required to derive symmetric key.

Request:
- `pc_dh_public_key`: base64 (PC ephemeral DH public key)
- `pc_nonce`: base64

Response 200:
- `mobile_dh_public_key`: base64 (mobile ephemeral DH public key)
- `mobile_nonce`: base64
- `kdf_context`: string

Notes:
- `pin_code` MUST NOT appear in request or response.
- Both sides derive `session_symmetric_key` from DH shared secret + nonces + `kdf_context`.
- Once `session_symmetric_key` is derived, pre-trust HTTP JSON bodies MUST use a trust-session envelope until `/trust/confirm` completes and pinned TLS trust is established.
- This endpoint is part of first-share trust bootstrap and does not require mTLS in this phase.

Trust-session encrypted envelope:
- `schema`: `dtis.instant-share.trust-envelope.v1`
- `nonce`: base64url AES-GCM nonce
- `ciphertext`: base64url AES-GCM ciphertext over the logical JSON body
- HTTP headers remain in cleartext; body payload is encrypted.

## 5.2 POST /api/instant-share/v1/trust/apply
Sent by PC to mobile after handshake. Carries PIN-related trust apply payload encrypted with `session_symmetric_key`.

Request:
- Entire JSON body is wrapped in the trust-session encrypted envelope after `/trust/handshake` succeeds.
- Decrypted logical request fields:
  - `flow_id`: `instant_share`
  - `payload_class`: `text | image`
  - `target_intent`: `clipboard_only | clipboard_or_file`
  - `trust_mode`: `first_share | trusted_direct`
  - `encrypted_payload`: base64
  - `encryption_alg`: string
  - `key_id`: string optional

Decrypted payload (logical fields):
- `pin_code`: string
- `mobile_display_proof`: string optional

Response 202:
- Response MAY be plaintext or MAY use the trust-session encrypted envelope during the pre-TLS trust phase.
- Decrypted logical response field:
  - `apply_status`: accepted

## 5.3 POST /api/instant-share/v1/trust/confirm
Sent by PC simultaneously with `/trust/apply` phase. This is a long-poll style confirm call to mobile service that waits for mobile-side user confirmation.

Request:
- Entire JSON body is wrapped in the trust-session encrypted envelope after `/trust/handshake` succeeds.
- Decrypted logical request fields:
  - `flow_id`: `instant_share`
  - `payload_class`: `text | image`
  - `target_intent`: `clipboard_only | clipboard_or_file`
  - `trust_mode`: `first_share | trusted_direct`
  - `pc_public_key_pem`: string

Behavior:
- Endpoint blocks (within timeout bound) until mobile confirms both devices show same PIN.

Response 200:
- Response MAY use the trust-session encrypted envelope during the pre-TLS trust phase.
- Decrypted logical response fields:
  - `mobile_public_key_pem`: string
  - `trust_status`: trusted

Failure:
- `409` with `PIN_MISMATCH_OR_REJECTED` when user rejects or PIN mismatch occurs.

Trust relationship is established when:
- `/trust/apply` accepted, and
- `/trust/confirm` returns success with mobile public key.

Pinning rule after trust confirmation:
- PC MUST persist `mobile_public_key_pem` returned by `/trust/confirm` for the active session.
- All subsequent transfer HTTPS calls (`/payload/*` and `/delivery-result`) MUST validate that the mobile HTTPS server certificate presents the same public key as `mobile_public_key_pem`.

## 6. Payload Download Endpoints

Caller/host direction:
- PC calls these endpoints on the mobile-hosted HTTP service using session bootstrap transport data.
- After trust completes, mobile remains the source of truth for the shared payload and desktop downloads the payload from the iOS-hosted server.
- Sensitive payload-download calls SHOULD include valid session signature headers and session-id matching.
- Payload-download and delivery-result calls MUST use HTTPS with public-key pinning derived from `mobile_public_key_pem` returned by `/trust/confirm`.

## 6.1 POST /api/instant-share/v1/payload/text
Used when `payload_class = text` and PC needs to fetch the shared text body.

Request:
- metadata envelope only:
  - `flow_id`
  - `payload_class`
  - `target_intent`
  - `trust_mode`

Response 200:
- `state`: `delivering`
- `text_utf8`: string

## 6.2 POST /api/instant-share/v1/payload/image
Used when `payload_class = image` and PC needs to fetch the shared image bytes.

Request:
- metadata envelope only:
  - `flow_id`
  - `payload_class`
  - `target_intent`
  - `trust_mode`

Response 200:
- raw response body: image bytes
- `Content-Type`: image MIME type
- `X-Instant-Share-Filename`: optional suggested filename
- `X-Instant-Share-Manifest`: optional JSON string with image metadata

Abort behavior:
- There is no dedicated `/abort` endpoint in v1.
- User abort is represented by transport termination from caller side and finalized through delivery-result state reporting.

## 7. Delivery Result Endpoint

## 7.1 POST /api/instant-share/v1/delivery-result
Reports final delivery outcome.

Request:
- `state`: `done | failed | timed_out | aborted`
- `target_result`:
  - `clipboard_written`: boolean
  - `files_written_count`: integer
  - `output_paths`: array of strings
- `error_code`: optional
- `error_message`: optional

Response 200:
- `ack`: true

## 8. State Machine

States:
- `bootstrapped` (after HTTP session bootstrap POST completes)
- `negotiating`
- `transferring`
- `delivering`
- `done | failed | timed_out | aborted`

Allowed transitions:
- `bootstrapped -> negotiating`
- `bootstrapped -> transferring` (trusted-direct path when mDNS TXT `signature` verification succeeds)
- `negotiating -> transferring`
- `transferring -> delivering`
- `delivering -> done`
- `negotiating -> failed`
- `transferring -> failed`
- `transferring -> timed_out`
- `transferring -> aborted`
- `delivering -> failed`

Rules:
- Only one active session (`bootstrapped|negotiating|transferring|delivering`) per receiver.
- Additional attempts during active session return `RECEIVER_BUSY_SINGLE_SESSION`.
- `negotiating` MAY be skipped when mobile verifies PC mDNS TXT `signature` and trust mode is `trusted_direct`.
- When `negotiating` is skipped, trust endpoints are not required for that session.

## 9. Error Model

Standard error response:
- `error_code`: string
- `message`: string
- `correlation_id`: string
- `retryable`: boolean
- `details`: object optional

Defined error codes:
- `RECEIVER_BUSY_SINGLE_SESSION`
- `SESSION_ID_MISMATCH`
- `TARGET_INTENT_INVALID_FOR_PAYLOAD`
- `HANDSHAKE_REQUIRED`
- `PIN_MISMATCH_OR_REJECTED`
- `CONFIRM_TIMEOUT`
- `SIGNATURE_VERIFICATION_FAILED`
- `SESSION_SIGNATURE_INVALID`
- `TRUSTED_KEY_NOT_FOUND`
- `TLS_PIN_VALIDATION_FAILED`
- `PAYLOAD_UNREADABLE`
- `DELIVERY_PATH_INVALID`
- `TRANSFER_TIMEOUT`
- `USER_ABORTED`

## 10. Security Requirements

- First share:
  - MUST complete `/trust/handshake`.
  - MUST derive symmetric key from DH output and handshake context.
  - MUST send PIN only via encrypted `/trust/apply` payload.
  - MUST complete `/trust/confirm` with pc/mobile public key exchange.
- Trusted direct share:
  - MUST verify mDNS TXT `signature` record.
  - MUST enforce certificate pin validation during TLS negotiation.
  - MUST verify `X-Session-Signature` against trusted PC public key and `X-Session-Id`.
- After `/trust/confirm`, PC MUST use the returned `mobile_public_key_pem` as the HTTPS public-key pin for transfer calls in the active session.
- Any PIN mismatch/rejection MUST terminate trust flow.
- All HTTP calls MUST carry `X-Session-Id`; receiver MUST enforce match with HTTP bootstrap session.
- Request-level replay hardening beyond session-id signature verification (for example, monotonic counters) is intentionally deferred to a later pass.

## 11. Observability Contract

Every endpoint must emit telemetry attributes:
- `instant_share.session_id`
- `instant_share.correlation_id`
- `instant_share.payload_class`
- `instant_share.target_intent`
- `instant_share.trust_mode`
- `instant_share.state`
- `instant_share.error_code` (if any)

## 12. Compatibility and Rollout

- Feature flag gate required on desktop receiver and mobile sender.
- Unsupported protocol version response:
  - `426 Upgrade Required`
  - `error_code: PROTOCOL_VERSION_UNSUPPORTED`

## 13. Acceptance Checklist for Task 1.1

Task 1.1 is complete when:
- mDNS service advertisement (`_instantshare._tcp` with TXT records) is implemented in desktop daemon.
- Session bootstrap via HTTP POST to PC endpoint is implemented (no `/sessions` endpoint).
- Trust endpoints `/trust/handshake`, `/trust/apply`, and `/trust/confirm` are implemented with specified cryptographic behavior.
- Payload-download and result endpoints in sections 6 and 7 are implemented.
- Session/header contract and single-session rules are enforced.
- Error model and security requirements are covered by integration tests.
