# Instant Sharing Protocol API Spec

> **Status: Historical reference — protocol direction inverted.**  
> This spec was written for the original architecture where iOS hosted all endpoints and PC acted as the HTTP client.  
> The architecture was inverted so **PC hosts all endpoints and iOS acts as the HTTP client**.  
> See `openspec/changes/pc-hosted-trust-and-upload/` for current protocol contracts.  
> Key deviations from this spec documented inline below.

## 1. Scope

This document defines the dedicated instant-sharing protocol contracts for mDNS (Bonjour) discovery, session bootstrap, trust establishment, and payload transfer. The protocol is independent from QR backup session endpoints and MUST NOT depend on backup-session capability exchange.

Goals covered:
- Discover candidate PCs via desktop mDNS daemon service advertisement.
- Bootstrap transport with mobile-selected session ID plus mobile IP/port list over HTTP callback to PC.
- Establish first-time trust using DH key exchange, encrypted PIN apply, and confirm exchange.
- Transfer payloads over HTTPS using pinned trust and request authentication based on session-id signature verification.
- Enforce single active receive session per desktop receiver.

Out of scope:
- Backup-session APIs.
- Large-media optimization strategies.
- Concurrent instant-share transfers.

## 2. Transport, Versioning, and Session Bootstrap

> **Updated by `pc-hosted-trust-and-upload` change:**
> - Scheme is plain HTTP (HTTPS deferred).
> - Session-signature authentication deferred.
> - No AuBackup handoff — Share Extension handles full flow natively.
> - No dedicated bootstrap endpoint — bootstrap data (mobile_port, mobile_ip_list, payload_class, target_intent) merged into `/trust/handshake` body.
> - Direction inverted: **PC hosts** all endpoints on port 9527, **iOS calls** PC's endpoints as HTTP client.

- Base scheme: HTTP (HTTPS with self-signed certs deferred).
- API prefix: `/api/instant-share/v1`.
- Control plane content type: `application/json`.
- Binary transfer: `multipart/form-data`.
- Transport authentication:
  - First share uses the DH/PIN trust flow encrypted with session-key AES-GCM trust envelopes over HTTP.
  - Subsequent shares with session-id signature verification — **deferred**.
- Session bootstrap rule:
  - The Share Extension presents the mDNS-discovered device list.
  - When the user selects a PC, the extension calls `/trust/handshake` with bootstrap data merged into the body.
  - PC creates the session from the handshake body (no separate bootstrap endpoint).
- Role direction in v1:
  - **PC hosts** the instant-sharing HTTP service on port 9527.
  - **iOS acts as HTTP client** for trust and payload-upload operations.

Versioning:
- Backward-compatible changes: additive optional fields.
- Breaking changes: new major path (`/v2`).

## 3. Identity and Metadata

### 3.1 Required HTTP Headers

- `X-Instant-Share-Version`: protocol version (e.g., `1.0`).
- `X-Correlation-Id`: UUID generated per share attempt.
- `X-Session-Id`: UUID generated by Share Extension for the session.
- `X-Device-Id`: caller device identifier.
- `X-Session-Signature`: base64 signature of `X-Session-Id` using caller private key. *(Deferred — not enforced in current implementation.)*
- `X-Session-Signature-Alg`: signature algorithm identifier (for example `ed25519`). *(Deferred.)*

### 3.2 Required Metadata Fields

- `flow_id`: fixed value `instant_share`.
- `payload_class`: current implementation slice supports `text | image`.
- `target_intent`:
  - `text`: `clipboard_only`
  - `image`: `clipboard_or_file`
- `trust_mode`: `first_share | trusted_direct`.

## 4. mDNS Advertisement Contract (Desktop Daemon)

Desktop daemon MUST advertise one Bonjour service for discovery.

Service:
- Type: `_instantshare._tcp`
- Domain: `local`
- Name: human-readable device name (e.g. `My Mac`)
- Port: PC's instant-share HTTP API port

TXT Records:

| Key | Value | Description |
|-----|-------|-------------|
| `ver` | string | Protocol version (e.g. `1`) |
| `device_name` | string | Human-readable PC display name |
| `device_id` | string | Persistent unique device identifier (UUID) |
| `signature` | base64 | Cryptographic signature for trusted-direct verification |
| `signature_key_id` | string | Key identifier for signature verification |
| `timestamp_ms` | integer string | Unix millisecond timestamp for signature freshness |

Rules:
- PC MUST update `signature` and `timestamp_ms` periodically.
- Mobile MUST verify `signature` against pinned trusted public key before trusted-direct share.
- If signature verification fails, mobile MUST fall back to full first-share trust flow.

## 4b. Session Bootstrap Endpoint (PC HTTP API) — **REMOVED**

> Bootstrap endpoint was removed in `pc-hosted-trust-and-upload` change.  
> Bootstrap data (mobile_port, mobile_ip_list, payload_class, target_intent, trust_mode) is now embedded in the `/trust/handshake` request body.  
> Session creation happens on the PC when iOS calls `/trust/handshake` — no separate POST required.

## 5. Trust Endpoints

> **Direction inverted by `pc-hosted-trust-and-upload` change:**  
> All trust endpoints are now called **by iOS → PC** (not PC → iOS).  
> `/trust/confirm` is a simple POST (not long-poll) — iOS sends confirmation after user taps Confirm.  
> No X509 public key exchange — trust is session-key-only (HTTPS/X509 cert exchange deferred).  
> See `openspec/changes/pc-hosted-trust-and-upload/specs/pc-trust-endpoints/spec.md` for current contracts.

## 5.1 POST /api/instant-share/v1/trust/handshake
Performs DH handshake and returns material required to derive symmetric key.

Request (iOS → PC):
- `mobile_dh_public_key`: base64 (mobile ephemeral DH public key)
- `mobile_nonce`: base64
- `mobile_port`: integer (merged bootstrap data)
- `mobile_ip_list`: array of strings
- `payload_class`, `target_intent`, `trust_mode`: strings

Response 200 (PC → iOS):
- `pc_dh_public_key`: base64 (PC ephemeral DH public key)
- `pc_nonce`: base64
- `kdf_context`: string

Notes:
- `pin_code` MUST NOT appear in request or response.
- Both sides derive `session_symmetric_key` from DH shared secret + nonces + `kdf_context`.
- Once `session_symmetric_key` is derived, pre-trust HTTP JSON bodies MUST use a trust-session envelope until `/trust/confirm` completes.
- No session-signature verification, no mTLS, no X509 exchange in this phase.

Trust-session encrypted envelope:
- `schema`: `dtis.instant-share.trust-envelope.v1`
- `nonce`: base64url AES-GCM nonce
- `ciphertext`: base64url AES-GCM ciphertext over the logical JSON body
- HTTP headers remain in cleartext; body payload is encrypted.

## 5.2 POST /api/instant-share/v1/trust/apply
Called by iOS after handshake. PC generates PIN, encrypts it, returns to iOS.

Request (iOS → PC):
- Entire JSON body is wrapped in the trust-session encrypted envelope.
- Decrypted logical request fields:
  - `action`: `"request_pin"`

Response 202 (PC → iOS):
- Encrypted trust envelope containing:
  - `pin_code`: string (6-digit PIN)
  - `apply_status`: `"accepted"`

Semantics:
- PC generates the 6-digit PIN on receiving this request.
- PC displays the same PIN on its mini window.
- Both sides show the same PIN for user visual verification.

## 5.3 POST /api/instant-share/v1/trust/confirm
Called by iOS after user taps Confirm. Simple POST — no long-poll.

Request (iOS → PC):
- Encrypted trust envelope with plaintext:
  - `action`: `"confirm"`
  - `pin_verified`: `true`

Response 200 (PC → iOS):
- Encrypted trust envelope containing:
  - `trust_status`: `"trusted"`

Semantics:
- **Mobile-side only confirmation** — PC does not require a separate user action.
- No long-polling — iOS sends after user acts.
- No X509 public key exchange in this phase.

Failure:
- `409` with `PIN_MISMATCH_OR_REJECTED` when PIN mismatch occurs.

Trust relationship is established when:
- `/trust/apply` accepted, and
- `/trust/confirm` returns `trust_status: "trusted"`.

## 6. Payload Upload Endpoints

> **Direction inverted by `pc-hosted-trust-and-upload` change:**  
> PC now hosts upload endpoints; iOS uploads payloads TO PC.  
> See `openspec/changes/pc-hosted-trust-and-upload/specs/pc-upload-endpoints/spec.md` for current contracts.

## 6.1 POST /api/instant-share/v1/transfer/text
iOS uploads shared text to PC.

Request (iOS → PC):
- JSON body:
  - `text_utf8`: string
  - `metadata`: object (optional)

Response 200:
- `state`: `"delivered"`

## 6.2 POST /api/instant-share/v1/transfer/image
iOS uploads shared image bytes to PC.

Request (iOS → PC):
- Binary body with:
  - `Content-Type`: image MIME type
  - `X-Instant-Share-Filename`: original filename

Response 200:
- `state`: `"delivered"`

Abort behavior:
- There is no dedicated `/abort` endpoint in v1.
- User abort is represented by transport termination from caller side.

## 7. Delivery Result

Delivery result is processed locally on PC after upload completes. No separate endpoint — PC's `TransferHandler` stores the result internally and passes `output_file_path` via `TransferResult` for the mini window's "Show in Finder" action.

## 8. State Machine

States:
- `bootstrapped` (after HTTP session bootstrap POST completes)
- `negotiating`
- `transferring`
- `delivering`
- `done | failed | timed_out | aborted`

Allowed transitions:
- `bootstrapped -> negotiating`
- `bootstrapped -> transferring` (trusted-direct path when mDNS TXT `signature` verification succeeds)
- `negotiating -> transferring`
- `transferring -> delivering`
- `delivering -> done`
- `negotiating -> failed`
- `transferring -> failed`
- `transferring -> timed_out`
- `transferring -> aborted`
- `delivering -> failed`

Rules:
- Only one active session (`bootstrapped|negotiating|transferring|delivering`) per receiver.
- Additional attempts during active session return `RECEIVER_BUSY_SINGLE_SESSION`.
- `negotiating` MAY be skipped when mobile verifies PC mDNS TXT `signature` and trust mode is `trusted_direct`.
- When `negotiating` is skipped, trust endpoints are not required for that session.

## 9. Error Model

Standard error response:
- `error_code`: string
- `message`: string
- `correlation_id`: string
- `retryable`: boolean
- `details`: object optional

Defined error codes:
- `RECEIVER_BUSY_SINGLE_SESSION`
- `SESSION_ID_MISMATCH`
- `TARGET_INTENT_INVALID_FOR_PAYLOAD`
- `HANDSHAKE_REQUIRED`
- `PIN_MISMATCH_OR_REJECTED`
- `CONFIRM_TIMEOUT`
- `SIGNATURE_VERIFICATION_FAILED`
- `SESSION_SIGNATURE_INVALID`
- `TRUSTED_KEY_NOT_FOUND`
- `TLS_PIN_VALIDATION_FAILED`
- `PAYLOAD_UNREADABLE`
- `DELIVERY_PATH_INVALID`
- `TRANSFER_TIMEOUT`
- `USER_ABORTED`

## 10. Security Requirements (Current Implementation)

- First share:
  - MUST complete `/trust/handshake`.
  - MUST derive symmetric key from DH output and handshake context.
  - MUST send PIN only via encrypted `/trust/apply` payload.
  - MUST complete `/trust/confirm` (no public key exchange — deferred).
- Trusted direct share (deferred):
  - mDNS TXT `signature` verification, certificate pin validation, `X-Session-Signature` verification — all deferred.
- Any PIN mismatch/rejection MUST terminate trust flow.
- All HTTP calls MUST carry `X-Session-Id`; receiver MUST enforce match with active session ID.
- Transport is plain HTTP (no HTTPS) — security relies on AES-GCM trust envelope encryption for session-key-protected payloads.

## 11. Observability Contract

Every endpoint must emit telemetry attributes:
- `instant_share.session_id`
- `instant_share.correlation_id`
- `instant_share.payload_class`
- `instant_share.target_intent`
- `instant_share.trust_mode`
- `instant_share.state`
- `instant_share.error_code` (if any)

## 12. Compatibility and Rollout

- Feature flag gate required on desktop receiver and mobile sender.
- Unsupported protocol version response:
  - `426 Upgrade Required`
  - `error_code: PROTOCOL_VERSION_UNSUPPORTED`

## 13. Acceptance Checklist for Task 1.1

Task 1.1 is complete when:
- mDNS service advertisement (`_instantshare._tcp` with TXT records) is implemented in desktop daemon.
- Trust endpoints `/trust/handshake`, `/trust/apply`, and `/trust/confirm` are implemented with specified cryptographic behavior.
- Upload endpoints `/transfer/text` and `/transfer/image` are implemented.
- Session/header contract and single-session rules are enforced.
- Error model is covered by integration tests.
