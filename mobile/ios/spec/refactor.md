Refactor the data handling of the ios app in mobile/ios. Must conform to the following best pratices:
1. Data package. Only put domain scoped data items in an abstract data type. And each view or page or corresponding model access one or more data sources from each domaon. For example, we need a backup session that is persisted to disk, with fields { session id, backup_status, desktop name, updated at, etc }. Home page can use such data to show last backup info. And the transfer service should offer a transfer session contains { total item count, transfered count, real time transfer speed, etc } which should not be persisted. Transfer page can leverage this to show the transfer progress UI. The transports (usb or wifi or both) used during pairing & transfer, is transient state and should not be persisted either. One bad example of packaging is TransferSnapshot.guidanceMessage which should be derived from the backup status by view model instead of being crammed into that data structure. Check other string fields for such anti-pattern usage and fix them as well
2. Single source of truth. Only the domain module owns the data and offers data access to other modules via whichever methods appropriate - direct properties, async query methods or subscription for data changes. In our ios project, we have the backup session concept but not a backup service to own it. Consider adding a backup session provider which offers backup session data access and is depended on both UI (reading the data) and services like pairing service (writing the data).
3. Permission status should usually be queried using system api realtime instead of being stored in memory or disk, except for  the custom 'remove after backup flag' which can be read/write into the permission service to be shared between different pages (permission page and transfer page). Do not persist this flag in disk. Page local data can be stored in vm for reactive style binding and be cautious to  trigger an reload from view lifecycles
4. Use `Combine` framework for data binding but watch for scenarios when the data is updating too frequent like data transfer, use throttling for such scenario or use a timer to trigger rerender
5. Inject services into vm directly if needed, for UI rendering and interaction.
6. Ignore existing persisted data. Do not try to migrate existing data to new scheme. As long as the app does not crash when updating from old version, it's fine.
7. Be cautious about thread safe. And try not to use some versioned data scheme which is unnecessarily complex. Just use existing concurrent tools offered by ios or swiftui. Ensure the UI is responsive.
8. Some data like 'pendingItemCount' is obsolete and should be ignored by the way, which is not needed in  this product. For each other fields, examine if there are obsolete  before deciding refactoring  them or just ignore them
9. If some data can be derived, do not store it in memory or disk.
