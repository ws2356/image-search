# Web-Based Instant Share (PC-to-Mobile Fallback Flow)

**Date:** 2026-07-05
**Status:** Implemented — verified pending E2E manual tests
**Author:** Brainstorming session (opencode + ws2356)

## Purpose

When a user scans an instant-share QR code and the native InstantShare iOS app
is NOT installed, the QR link opens in the mobile browser. This design
describes a web fallback flow that completes the pc-to-mobile transfer
entirely in the browser, using WebRTC for the local-network data path and a
CDN-hosted WebSocket relay for signaling.

The existing native-only flow is untouched. The same QR URL
(`https://dl.boldman.net/share?…`) is reused; the universal link continues to
intercept it on devices with the native app installed, and the web SPA loads on
devices without it.

## Constraints & Context

- The QR encodes `ips`, `p` (http port), `sp` (tls port), `sid` (session id),
  and `opt` (6-digit opt code). These are already generated by the PC's
  `QRTriggerHandler` + `QRTriggerMiniWindow`.
- The PC's local HTTPS server uses a self-signed device cert generated at
  runtime. A CDN-origin page (`https://dl.boldman.net`) cannot `fetch()` to a
  self-signed endpoint — browsers block it without UI override — and a plain
  HTTP `http://<lan-ip>:<port>` target is blocked as mixed content.
- WebRTC `RTCDataChannel` uses DTLS with self-generated certs that are not
  validated against any CA, sidestepping the self-signed problem entirely. On
  the same LAN, ICE resolves to direct host candidates — no STUN/TURN needed.
- WebRTC still needs **out-of-band signaling** (offer / answer / ICE exchange).
  The QR already contains the session id; a tiny CDN-hosted WebSocket relay
  pairs the PC and browser by `sid`.

## Design decisions (locked)

| Decision | Choice | Rationale |
|---|---|---|
| Signaling transport | CDN WebSocket relay (`wss://dl.boldman.net/relay`) | Clean cross-network signaling; existing local endpoints untouched; minimal infra |
| Trust & crypto on the data channel | opt_code-as-token on DTLS only — skip DH handshake and X509 cert exchange | DTLS already secures the channel; opt_code (from QR) is the single auth gate; browser can't persist native certs for revisit anyway |
| PC WebRTC peer | `aiortc` inside the existing asyncio runtime | One language, one process; reuses `QRTriggerHandler` + `TrustSessionRegistry` in-process; no new binary |
| CDN relay | Tiny Node.js/Bun relay shipped in `web/instant-share/relay/` | Same JS toolchain as the web build; ~100 lines; ephemeral rooms keyed by `sid` |
| Web delivery UX | Type-aware auto-delivery (copy for text, open for link, render for html, download/save for files) — auto-receives after auth, no user button | Matches existing per-payload-class rules within browser capabilities; mirrors native auto-flow |
| Wire-protocol naming | Hide "stash" concept; speak only in `session_id` + file `index` | The stash is a PC/macos-share-extension implementation detail; the web client vocabulary stays minimal |
| Local dev workflow | Skip — deploy directly to the boldman.net remote for testing | Faster iteration for this project's single-host deployment model |

## Architecture

### New components

| Component | Location | Responsibility |
|---|---|---|
| Web receiver SPA | `web/instant-share/` (Vite + React + TS + Tailwind, native browser WebRTC) | Parses QR params, joins relay room, exchanges SDP/ICE, opens data channel, authenticates with opt_code, auto-fetches manifest + files, delivers type-aware UX (copy / download / save-to-photos) |
| CDN WebSocket relay | `web/instant-share/relay/server.mjs` (~100-line Node/Bun) | Ephemeral room keyed by `session_id`; forwards offer / answer / ICE / leave between the two peers. No state, eviction on disconnect / TTL |
| PC WebRTC peer | `dt_image_search/instant_sharing/webrtc_peer.py` (`aiortc`, inside the existing asyncio runtime) | On QR-stash creation, dials the wss relay, builds `RTCPeerConnection`, opens a data channel, authenticates opt_code, then serves manifest/files by calling the existing `QRTriggerHandler` in-process |

### Touch points with existing code (minimal)

- **Reuse:** `QRTriggerHandler` (stash, opt_code, expiry, claim),
  `TrustSessionRegistry.get_session(sid).verify_opt()` for opt verification +
  internal `stash_id` lookup, `QRTriggerMiniWindow` QR generation (URL
  unchanged), `telemetry_client.log` / `add_span`.
- **New glue:** `QRTriggerHandler._on_stash_created` (already exists) spawns
  the WebRTC peer in the existing asyncio loop. `_on_stash_claimed` and
  `_on_stash_expired` (already exist) tear down the peer. **Note:** text
  stashes have content inline in the manifest and never trigger
  `retrieve_stash_file`, so the WebRTC peer must explicitly invoke the claim
  path (`_on_stash_claimed` callback + `StashEntry.claimed = True`) after a
  text-only manifest is acknowledged by the browser, mirroring the claim
  side-effect `retrieve_stash_file` performs for file stashes.

### End-to-end flow

```
PC                                          CDN relay                          Mobile browser
────                                        ─────────                          ──────────────
1. User triggers share (file/text)
2. QRTriggerHandler.handle_trigger()
   → creates stash (PC-internal) + session_id + opt_code
   → _on_stash_created fires
   → webrtc_peer.start(session_id, opt_code)
   → opens wss to relay?sid=<session_id>    3. relay creates room <sid>
4. QRTriggerMiniWindow shows QR
   (URL unchanged)
                                                 5. Native app not installed
                                                    → browser opens URL
                                                    → Vite SPA loads
                                                    → reads ?sid=&opt=
                                                    → opens wss to relay?sid=<sid>
                                                    → relay pairs PC + browser in <sid>
6. aiortc creates RTCPeerConnection + data channel
   → sends offer via wss                        7. relay forwards offer → browser
                                                8. browser creates answer → sends via wss
9. relay forwards answer → aiortc
10. ICE candidates trickle via wss both ways
11. DTLS handshake → data channel "instant-share" open
    ┌────────── DC OPEN ──────────────────────────────────┐
12. browser → PC: {"msg":"auth","opt_code":"<opt>"}
13. PC verifies opt via TrustSessionRegistry.get_session(sid).verify_opt(opt)
    → internally resolves session → stash (PC-only; invisible to browser)
14. PC → browser: {"msg":"auth_ok","session_id":"<sid>","file_count":N,"payload_type":"file"|"text"|"link"|"html"}
    Or {"msg":"auth_error","error":"invalid_opt"} → PC closes DC
15. browser → PC: {"msg":"manifest"}            (auto-sent on auth_ok)
16. PC → browser: {"msg":"manifest","files":[{index,type,filename?,content_type,size_bytes?}]}
    (internally calls retrieve_stash_content — hidden)
17. For each index i (serial, auto-triggered):
    browser → PC: {"msg":"download","index":i}
    PC → browser: {"msg":"file_start","index":i,"content_type":"…","filename":"…","size":N}
    PC → browser: <ArrayBuffer: 64KB chunk frames, may repeat>
    PC → browser: {"msg":"file_end","index":i}
    (internally calls retrieve_stash_file — hidden)
    → on first successful download, internally marks claimed
    → fires _on_stash_claimed → QRTriggerMiniWindow "Delivered"
18. browser renders/delivers per type (copy/download/save) — auto for non-iOS-Safari; button tap on iOS Safari for clipboard
19. Either side: {"msg":"bye"} or wss close
20. relay evicts room
```

### Invariants

- Only `sid` and `opt` QR params are required by the web app. `ips`/`p`/`sp`
  are preserved for the native app and ignored by the web flow.
- WebRTC has the PC as offerer, browser as answerer. Single data channel
  named `instant-share`, opened once.
- Relay sees only SDP / ICE / leave messages. Never payloads, never
  opt_code (opt_code is sent over the DTLS-protected DC).
- Single active session per receiver (existing invariant preserved).
- The wire protocol never mentions `stash_id`. The WebRTC peer looks up the
  stash internally via `TrustSessionRegistry.get_session(sid).stash_id`.

## Wire protocol

All control messages are JSON strings. File payloads are raw `ArrayBuffer`
binary frames. Single data channel `instant-share` with
`binaryType = "arraybuffer"`. Browser distinguishes by `event.data instanceof
ArrayBuffer` (binary) vs `typeof event.data === "string"` (control JSON).

### Relay signaling (wss JSON — not over DC)

| Direction | Message | Purpose |
|---|---|---|
| client → relay | `{"type":"join","sid":"<uuid>","role":"pc"\|"browser"}` | Join room; role assigned by explicit field |
| relay → client | `{"type":"joined","sid":"<uuid>","role":…}` | Confirm join |
| — | (any other message) | Relay forwards verbatim to opposite-role peer |
| client → relay | `{"type":"leave"}` | Graceful exit |
| relay → client | `{"type":"peer_left"}` | Other peer disconnected |
| relay → client | `{"type":"room_full"}` | Second joiner for same role rejected |

Relay is dumb: forwards every non-`join` message to the other peer in the
room. No parsing, no validation beyond `sid` routing. Eviction: `leave`,
socket close, or 5-minute idle TTL.

### Data channel control messages (JSON UTF-8 strings)

| Direction | Field | Purpose |
|---|---|---|
| browser → PC | `{"msg":"auth","opt_code":"123456"}` | First DC message; authenticate |
| PC → browser | `{"msg":"auth_ok","session_id":"<uuid>","file_count":N,"payload_type":"file"\|"text"\|"link"\|"html"}` | Auth success + metadata |
| PC → browser | `{"msg":"auth_error","error":"invalid_opt"}` | Failed auth → PC closes DC |
| browser → PC | `{"msg":"manifest"}` | Auto-sent on auth_ok |
| PC → browser | `{"msg":"manifest","files":[{"index":0,"type":"text","content_type":"text/plain","size_bytes":42,"content":"…"},…]}` | Text entries inline `content`; file entries have `filename` instead |
| browser → PC | `{"msg":"download","index":i}` | Auto-sent serially per file |
| PC → browser | `{"msg":"file_start","index":i,"content_type":"image/png","filename":"photo.png","size":N}` | Announce binary frames |
| PC → browser | `<ArrayBuffer>` one or more 64KB chunks | Raw file bytes |
| PC → browser | `{"msg":"file_end","index":i}` | End of binary for index i |
| PC → browser | `{"msg":"error","code":"expired"\|"not_found"\|"busy","message":"…"}` | Upstream stash/session error |
| PC → browser | `{"msg":"bye"}` | Graceful close (PC tears down 60s after done) |
| either → either | `{"msg":"bye"}` | Graceful DC close |

### Ordering guarantees

- `auth` is first DC message from browser. PC rejects any other first message
  with `{"msg":"error","code":"not_authorized"}` and closes DC.
- `manifest` auto-sent immediately after `auth_ok`.
- `download` requests serial — browser waits for each `file_end` before
  sending the next `download`.
- Text / link / html payloads are inline in the `manifest` message; no
  separate `download` round-trip — browser delivers them immediately on
  manifest receipt.
- `bye` triggers `done` UI state on browser.

### State machine (`App.tsx`)

```
booting → connecting → authenticating → transferring → done
                                  ↘ error (expired/busy/auth_error/disconnected)
```

- **booting:** parse URL, validate `sid`+`opt` exist. Missing → error.
- **connecting:** open wss, join room, exchange SDP/ICE, await DC open.
- **authenticating:** send `auth`, await `auth_ok`. 15s timeout → error.
- **transferring (auto):** on `auth_ok`, auto-send `manifest`, iterate
  indices serially, deliver text payloads inline from manifest, download
  file payloads via `file_start`/ArrayBuffer/`file_end`. Update per-file
  progress.
- **done:** show success + per-item re-copy / open / save buttons.

No user-facing "Receive" button — the flow mirrors the native app's
auto-receive UX.

### `deliverer.ts` rules

| `type` | Action |
|---|---|
| `text` | `navigator.clipboard.writeText(content)` + toast. iOS Safari exposes a "Copy" button on `DoneScreen` (clipboard requires user gesture). |
| `link` | Render clickable `<a target="_blank">` + "Copy" button. |
| `html` | Render in sandboxed `<iframe srcdoc=…>` + "Copy" button. |
| `file` / `image` | Build `Blob` from accumulated `ArrayBuffer` chunks, auto-trigger `<a download>`. For images, also expose "Save to Photos" via `navigator.share({files:[…]})` where supported (iOS 15+, user gesture). |

## CDN WebSocket relay (`relay/server.mjs`)

~100-line Node.js (or Bun) process. Deployed as a long-running systemd service
on the boldman.net host; nginx reverse-proxies `wss://dl.boldman.net/relay`
→ `127.0.0.1:8787`.

Responsibilities:
- Maintain `Map<sid, {pc?: WebSocket, browser?: WebSocket, createdAt: number}>` in memory.
- On `join`: store peer under explicit `role`. Reject second joiner of the
  same role with `{"type":"room_full"}`.
- Forward every other message to the opposite-role peer (verbatim passthrough).
- Eviction: on `leave`, on socket close, or 5-minute idle TTL — close the
  other peer's socket and delete the room.
- No logging of message contents (only connection events).

## PC WebRTC peer (`dt_image_search/instant_sharing/webrtc_peer.py`)

```python
class WebRTCPeer:
    """Bridges a browser's WebRTC data channel to QRTriggerHandler stash
    logic, authenticating via opt_code on the DTLS-protected channel."""

    def __init__(
        self,
        *,
        session_id: str,
        opt_code: str,
        qr_trigger_handler: QRTriggerHandler,
        trust_session_registry: TrustSessionRegistry,
        relay_url: str,           # wss://dl.boldman.net/relay
        telemetry_correlation_id: str,
    ): ...

    async def start(self) -> None:
        """Dial relay, await browser join, send offer, negotiate DC."""

    async def _on_message(self, msg: str | bytes) -> None:
        """Dispatch: auth → manifest → download → bye."""

    async def stop(self) -> None:
        """Close DC, PC, wss. Idempotent."""
```

Lifecycle wiring:
- `QRTriggerHandler._on_stash_created` (existing callback) → spawn `WebRTCPeer` in the existing asyncio loop (uvicorn already provides one).
- `_on_stash_claimed` → `peer.stop()` + `QRTriggerMiniWindow.on_claimed`.
- `_on_stash_expired` → `peer.stop()` + `QRTriggerMiniWindow.on_expired`.

Internal handlers:
- `auth`: verify opt via `trust_session_registry.get_session(session_id).verify_opt(opt_code)`. On success, look up `stash_id` from the same `TrustSession` (internal only — never on the wire) and respond `auth_ok` with `file_count` + `payload_type`. On failure, send `auth_error` and `stop()`.
- `manifest`: call `qr_trigger_handler.retrieve_stash_content(stash_id)`; translate to wire `manifest` (strip `stash_id`).
- `download`: call `qr_trigger_handler.retrieve_stash_file(stash_id, index)`; if bytes, send `file_start` (string) → 64KB ArrayBuffer chunks → `file_end` (string). Backpressure: check `dc.bufferedAmount`; if > 16MB, `await asyncio.sleep(0)` until drained. On 404/410, send `{"msg":"error","code":"expired",…}`.

Concurrency: single peer per session. Relay rejects second browser join.

## Error handling & edge cases

### Error handling matrix

| Scenario | Detection | Browser UI | PC behavior | Relay behavior |
|---|---|---|---|---|
| Invalid QR params (missing sid/opt) | URL parse on boot | `ErrorScreen`: "Invalid link" | N/A | N/A |
| Relay unreachable (wss fail) | `WebSocket` onerror/onclose | `ErrorScreen`: "Cannot connect, try again" + retry button | PC peer marks session failed; `QRTriggerMiniWindow` shows "Expired" after 300s TTL | N/A |
| Browser joins but PC never connected | 15s timeout on browser waiting for offer | `ErrorScreen`: "PC not responding" | PC may join late; relay holds room until TTL | Relay holds room 5min |
| Auth failed (wrong opt_code) | PC sends `auth_error` | `ErrorScreen`: "Code invalid or expired" | `peer.stop()` + mark session failed + mini-window "Expired" | Evict on disconnect |
| Stash expired mid-transfer | `retrieve_stash_content/file` returns 410 | `ErrorScreen`: "Expired" with stale file count | `peer.stop()` + mini-window "Expired" | Evict on disconnect |
| DC disconnected mid-transfer | RTC `onclose` / `iceconnectionstatechange` failed | `ErrorScreen`: "Connection lost" + retry button (re-scan QR) | `peer.stop()`; if stash unclaimed, mini-window stays showing QR (re-claimable) | Evict on disconnect |
| Concurrent claim (2nd browser joins same sid) | Relay rejects 2nd `join` for same role | `ErrorScreen`: "Already in use" | N/A | Relay sends `room_full` to 2nd joiner |
| Large file (>50MB) | Streaming 64KB chunks | Progress bar updates; PC pauses on `bufferedAmount` high-watermark | Backpressure: if `dc.bufferedAmount > 16MB`, `await asyncio.sleep(0)` until drained | N/A |
| Text-only payload (no files) | manifest `file_count=0` with inline content | `TransferScreen` flashes → `DoneScreen` with "Copied" toast | PC explicitly invokes claim side-effect after manifest ack (text stashes never call `retrieve_stash_file`); then `peer.stop()` + mini-window "Delivered" | Evict on disconnect |
| Browser tab closed mid-transfer | wss close event on relay → forwarded as `peer_left` | N/A | If claimed, mini-window "Delivered"; if not, mini-window stays (re-claimable) | Evict room |
| PC app quits mid-transfer | Process death → wss close on relay | `peer_left` from relay → `ErrorScreen`: "PC disconnected" | N/A | Evict room |

### Edge cases

**Re-delivery on same session:** After `done`, the browser keeps the
`RTCDataChannel` open for 60s. User can re-tap copy / open / save on the
`DoneScreen`. After 60s, PC sends `bye` and tears down.

**Backpressure & memory:** PC streams files in 64KB ArrayBuffer chunks. Before
each chunk, check `dc.bufferedAmount`; if > 16MB, `await asyncio.sleep(0)`
until drained. Browser reassembles chunks into a Blob per index. No full-file
buffering in JS memory beyond the accumulated Blob (browser-managed, paged to
disk if large).

**Mixed content-safe:** Web app origin is `https://dl.boldman.net` (valid TLS).
Relay is `wss://dl.boldman.net/relay` (same origin). The `RTCDataChannel` uses
DTLS with self-generated certs (browser-internal, unvalidated against CA). No
mixed-content violations.

**Race: native app + browser both open the QR:** Universal link intercepts
first on native-installed devices. If both somehow connect, the PC's
single-session invariant kicks in: the first claimant wins, second gets
`error: busy`. Existing `SessionRegistry` already enforces this.

**Mobile Safari quirks:**
- `navigator.clipboard.writeText` requires a user gesture. Since auto-download
  is triggered by page load (not direct user tap), the copy step must wait for
  a user tap on iOS Safari. `DoneScreen` exposes a prominent "Copy" button as
  the universal fallback; Chrome / Android auto-copies after page load.
- `navigator.share({files})` for "Save to Photos" requires iOS 15+ and a user
  gesture; exposed as a button on `DoneScreen`, not auto-invoked.

8. Browser tab closed mid-transfer → PC mini-window stays claimable.

### Not tested in v1 (manual-only)

- ICE failure on restrictive networks (NAT loopback issues) — documented as
  known limitation; WebRTC host candidates work on same-LAN, but symmetric
  NAT / firewall may block. Fallback is native app.
- Concurrent multi-device scans — existing single-session invariant handles;
  not formally tested.

## Open questions / future work

- **TURN fallback:** If users hit restrictive NATs even on same-LAN (rare but
  possible with AP isolation / guest networks), a TURN server on boldman.net
  could relay DTLS traffic. Out of scope for v1; revisit if telemetry shows
  ICE failure rates > 2%.
- **Desktop browser support:** The flow is mobile-first (QR scan → phone
  browser), but desktop browsers can scan the QR into the same machine's
  browser for testing. Not a supported target for v1.
- **Browser-persisted trust for revisit:** v1 re-runs the full auto-flow on
  every scan. Future: persist a "trusted PC" flag per `(browser, sid-prefix)`
  in IndexedDB and skip the auth round-trip for revisits. YAGNI for v1.
